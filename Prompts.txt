You are a senior software architect who also has electronics hardware and ham radio experience.

You are asked to help architect a new project.

The project is to create an implementation of MeshCore (https://meshcore.co.uk/) on a Rapsberry PI 4 (or greater).

The Rapsberry PI has a LoRa (https://en.wikipedia.org/wiki/LoRa) expansion hat attached to it which contains two radio modules, one for the 434MHz spectrum and one for the 868MHz spectrum.

The LoRa modules are from Uptronics https://store.uputronics.com/products/raspberry-pi-loratm-expansion-board?variant=48647705624910.

The LoRa modules are the HopeRF RFM95W (868MHz) and RFM98W (434MHz).

The RFM95W uses the Semtech SX1276 chip (https://dwmzone.com/en/lora-wireless-modules/670-rfm95w-semtech-sx1276-868mhz-915mhz-hoperf-lora-transceiver-rf-module.html)

The RFM98W uses the Semtech SX1278 chip (https://dwmzone.com/ru/lora-wireless-modules/614-rfm96w-rfm98w-sx1276-sx1278-433mhz-470mhz-hoperf-lora-rf-module.html)

The datasheet for these modules is at: https://github.com/SeeedDocument/RFM95-98_LoRa_Module/blob/master/RFM95_96_97_98_DataSheet.pdf

The datasheet for the SX1276 and SX1278 chips is at: https://semtech.my.salesforce.com/sfc/p/#E0000000JelG/a/2R0000001Rbr/6EfVZUorrpoKFfvaF_Fkpgp5kzjiNyiAbqcpqh9qSjE

Both modules are connected over SPI.

The module in the CE0 slot is connected to DIO0/WPi6 and DIO5/WPi5 and is the 868MHz variant RFM95W.

The module in the CE1 slot is connected to DIO0/WPi27 and DIO5/WPi26 and is the 484MHz variant RFM98W.

The project is to create an implementation of MeshCore which can:
 * Act as a client
 * Act as a repeater

In both modes, it will have the following interfaces:
 * A command line interface similar to the meshtui (https://github.com/ekollof/meshtui) or meshcore-cli projects.
 * A Web interface, potentially using the FastAPI mechanism.
 * An exposed API which follows the standard MeshCore API used by https://github.com/meshcore-dev/meshcore_py

 The Over The Air protocol should follow that defined in the meshcore-dev project at: https://github.com/meshcore-dev/MeshCore

 The drivers for the RFM95W and RFM98W modules should draw inspiration from the various SX127* files in the src/helpers/radiolib directory of the MeshCore project (https://github.com/meshcore-dev/MeshCore/tree/main/src/helpers/radiolib)

 The immediate task is to complete an ai.md file so that it can be used by both human developers and AI tools to understand the project scope.

 A template has been prepared as an ai.md file and needs filling and completed.

 Please make suggestions to complete this file.

 Please do not implement any code at this time. The ai.md file is the only file in scope.

Context:

You are a Software Architect with Electronics and Radio experience.

You are developing a system which can use LoRa modules attached to a Raspberry PI to scan the radio frequencies to look for radio transmissions.



The Raspberry PI has up to two LoRa modules attached to the CE0 and, optionally, the CE1 lines of the SPI0 module.



Any attached module may be a RFM95W (High Frequency) or RFM98W (Low Frequency) module.



It is not possible for software to check what capabilities they have, but it is possible for software to check:

* Whether a module is attached to a specific SPI CE line.

* Whether that module can try to output high frequency.

* Whether that module can try to output low frequency.

* Whether that module can try to output both high and low frequencies.



The User will pass a configuration into the program and software is needed to determinine if that configuration is feasible and accept or reject that configuration accordingly.



For example, a configuration may say that there is a module attached to CE1, but we determine that there is no module attached to CE1 and therefore reject that configuration.



Or, a configuration may say that there is an RFM95W (High Frequency)  module attached to CE0 and we determine that:

a) There is a module attached to CE0 and

b) It is capable of multi-band (which includes high-frequency) output.



Therefore we accept this configuration.



Task:

* Please create a class which captures these capabilities in a way that allows a user to describe what they think is attached to a CE line on an SPI bus and also that allows the src/drivers/lora_detection.py file to test the configuration and accept or reject it.

* Please provide sufficient parameters to the "detect-modules" task in src/cli/check_hardware.py to allow the user to optionally create such a configuration and have the src/drivers/lora_detection.py class validate or reject that user-created configuration.

Constraints:

* All the hardware detection logic which is possible is already present in src/drivers/lora_module.py. Do not add any more.

* It has already been determined that none of the registers within the devices will reliably provide any different responses, so do not try to think of any additional ways that they can.
